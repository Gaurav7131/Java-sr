Q1: What are the SOLID principles in Java?

A: SOLID is an acronym for five object-oriented design principles intended to make software designs more understandable, flexible, and maintainable. They are: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, and Dependency Inversion.

Q2: Can you explain the Single Responsibility Principle (SRP)?

A: SRP states that a class should have one, and only one, reason to change. This means a class should only have one primary job or responsibility. For example, a class that handles database connections should not also be responsible for formatting the output to the user.

Q3: Why is the Single Responsibility Principle important?

A: It makes the code highly maintainable and easier to test. If a class has multiple responsibilities, changing one part of the code might accidentally break another part (high coupling). SRP ensures that changes are isolated, reducing the risk of introducing new bugs.

Q4: How do you identify if a class is violating SRP?

A: If you find yourself using words like "AND" when describing what a class does (e.g., "This class processes payments and sends email receipts"), it is likely violating SRP. Also, if a class has thousands of lines of code or requires imports from completely unrelated packages, it's a strong sign it's doing too much.

Q1: What is the Open/Closed Principle (OCP)?

A: The Open/Closed Principle states that software entities (classes, modules, methods) should be open for extension but closed for modification. This means you should be able to add new functionality without altering existing, tested code.

Q2: How do we achieve OCP in Java?

A: We primarily achieve OCP by using Abstraction (Interfaces and Abstract Classes) and Polymorphism. Instead of using tightly coupled if-else or switch statements to check for specific types, we program to an interface so that new behaviors can be added by simply creating new classes that implement the interface.

Q3: Why is violating OCP considered bad practice?

A: Modifying existing code introduces the risk of creating new bugs in already working and tested features. It also makes the class larger, harder to read, and tightly coupled, which violates the Single Responsibility Principle as well.

Q4: How can you quickly identify if a class is violating OCP?

A: If a class contains massive if-else or switch statements that check the type or state of an object to determine behavior (e.g., if (type == "A") doThis() else if (type == "B") doThat()), it is highly likely violating OCP. Any new type will force a modification of this logic.
